# Speaker Intent · 演讲意图与话术方向

> 本文件给"谁来讲这页"的人看：每页讲什么、怎么过渡、用多少时间、注意什么雷区。所有话术是**口语方向**，不是逐字稿——演讲者应自然表达，禁止照念。

---

## 全场节奏心法

- **14 分钟，20 页**：平均 42 秒/页，但实际不均匀——钩子页和结果页要慢，流程页和迭代页要快。
- **三个"慢点"**：P02（钩子）、P07（记忆与谦逊）、P17（准确率反差）——这三页是情绪支点，必须留足停顿。
- **一个"不许超时"**：P08 四层架构总览 30 秒带过，细节全在后面三页。
- **全场禁词**：综上所述、在此背景下、众所周知、值得一提、不可忽视——AI 议论腔一律不说。

---

## 序幕

### P01 封面（~20 秒）
**讲什么**：报项目名、副标题、一句话立意，然后立刻进下一页。
**话术方向**：「各位评委好，我今天答辩的项目是——2026 世界杯冠军预测 Agent。我给它的定义是：一个会记忆、会想象、知道边界的预测 Agent。线上地址和仓库都在屏幕上，待会儿大家可以边听边看。」
**过渡**：「先回答一个问题：全世界那么多预测题，为什么是世界杯？」
**雷区**：不要在封面页解释技术栈，浪费开场势能。

### P02 钩子：为什么是世界杯（~50 秒 · 慢点）
**讲什么**：用"三层不确定性"重新定义问题，把评委从"看模型"拉到"看一个处理复合不确定性的系统"。
**话术方向**：「大家可能觉得，预测世界杯嘛，不就建个模型打个分。但我想说，难的从来不是建模。难在哪？三件事：第一，真实会变——已经踢完的 18 场结果得锁住，赛况天天在变；第二，概率要叠——你要跑一万次完整赛程，每场都要算晋级路径；第三，必须实时——不能算一次就死，得每天自动重算。所以世界杯不是一道预测题，是 Agent 能力的终极试炼场：它必须在'已经发生的真实'和'还没发生的概率'之间，持续思考。」
**过渡**：「那这个 Agent 到底怎么思考的？我用五分钟讲清楚。」
**停顿点**：说完"难的从来不是建模"后停 1 秒，制造悬念。
**雷区**：不要在这里报数字（18场、10000次留到后面），钩子页只立概念。

## Act I · Agent 如何思考

### P03 Agent ≠ 模型（~40 秒）
**讲什么**：把"Agent"从营销词变成五个可检验的动作。
**话术方向**：「我先纠正一个可能的误解：这不是'模型套个前端'就叫 Agent。模型只回答一个问题——谁赢。但一个真正的 Agent 要回答四个问题：凭什么、能信几成、新赛果来了怎么办、怎么跟人解释。所以我把它拆成五种能力：记忆、学习、想象、表达、谦逊。后面几页我一个一个讲。」
**过渡**：「先看它整体怎么动起来的——六步思考流程。」
**雷区**：不要念五个能力的定义，让评委看图，你只点"模型 vs Agent"的差异。

### P04 六步思考流程（~45 秒）
**讲什么**：把 main.py 的 6 步日志讲成"一个思考者的六个动作"。
**话术方向**：「这就是我们后端 main.py 的六步。但我希望大家别把它当流水线看，它其实是一个思考过程：第一步加载 14 队档案和 395 场历史，这是'记住事实'；第二步构建 4 维特征，这是'形成直觉'；第三步训练随机森林，这是'学习历史'；第四步蒙特卡洛跑一万次，这是'想象未来'；第五步预测射手，这是'推演个体'；第六步输出 JSON，这是'开口说话'。六步不是管线，是一个会思考的过程。」
**过渡**：「我重点讲三步：直觉、想象、还有它的边界。」
**雷区**：不要逐个报数据规模（14队72场395场），挑"395 场历史"和"10000 次"两个说就够了。

### P05 特征工程：直觉从哪来（~45 秒）
**讲什么**：4 维特征是 Agent 的"判断依据"，并埋下 Act II 决策一的钩子。
**话术方向**：「Agent 的直觉从哪来？四件事：近期状态占 40%——最近踢得怎么样；硬实力占 30%——FIFA 排名加身价加历史夺冠；交锋 15%——历史谁克谁；情境 15%——东道主和大赛经验。但这里有个关键决策我先卖个关子：这四个权重，只用来给人看，模型训练的时候根本不用它们。为什么？这是 Act II 要讲的第一个决策。」
**过渡**：「先看它最'疯狂'的一步——想象一万种未来。」
**停顿点**：说"卖个关子"时手指向 P05 的金色决策标签，制造期待。

### P06 蒙特卡洛：想象一万种未来（~50 秒）
**讲什么**：100x 性能故事，同时证明想象力和工程力。
**话术方向**：「第四步是整个系统的核心：同一个赛程，跑一万遍，看谁拿冠军最多。听起来很爽，但朴素做法是 10000 次乘以 31 场，31 万次模型调用，要跑一分多钟，CI 里根本跑不动。我们做了个优化：14 支存活球队两两组合，只有 91 组对阵，先把每组的胜率预计算出来，模拟里只查表。结果调用次数从 31 万压到 496 次，提速 100 倍以上，从一分多钟降到两秒以内。我想说的是：不是算得慢就认真，预计算让'想象一万次'变成可以每天跑的日常事。」
**过渡**：「但光会想象还不够，一个靠谱的 Agent 还得有两样东西：记忆，和谦逊。」
**雷区**：数字别念太快，"31 万到 496"要咬清楚，这是全场最高光的工程数字。

### P07 记忆与谦逊（~50 秒 · 慢点）
**讲什么**：全场最人感的一页，给 Agent 注入"人格"。
**话术方向**：「这一页是我最想讲的。先说记忆：我们已经踢完的 18 场，真实结果会锁死在模拟里，剩下的才推演。这意味着这个系统不是赛前预测一次就完了，它是赛中实时预测，越往后越准。再说谦逊：我们给胜率设了一个 85% 的硬上限，再强也不说自己稳赢，超出的部分按比例退给平局和对手。为什么是 85 不是 90？因为我们不想让模型在强队身上过度自信。一句话：记忆让它随赛况进化，谦逊让它不翻车。我觉得这才像一个 Agent 该有的样子。」
**过渡**：「Agent 怎么思考讲完了。但思考归思考，工程上撑得住吗？看架构。」
**停顿点**：说完"不翻车"后停 2 秒，让"人格感"落地。

## Act II · 架构如何支撑

### P08 四层架构总览（~30 秒 · 快带过）
**讲什么**：给骨架图，30 秒带过，把时间留给后三页的决策。
**话术方向**：「这是我们的四层架构：数据层管事实、预测引擎层管思考、可视化层管翻译、部署层管让它活着。引擎层有 7 个后端模块。但我今天不想讲'是什么'，我想讲'为什么这么搭'——三个关键决策。」
**过渡**：「第一个决策：模型和可解释性，为什么要分开。」
**雷区**：不要逐层念职责，30 秒必须翻页。

### P09 决策一：模型与可解释性分离（~45 秒）
**讲什么**：讲清"为什么展示权重 ≠ 训练权重"的取舍。
**话术方向**：「刚才卖的那个关子，在这里解开。如果直接用 4 维加权去训练模型，会有两个问题：一是模型会过拟合这几个权重，二是黑箱不可解释。我们选了更复杂的方案 B：随机森林吃 10 维原始特征去训练，4 维权重只用来生成中文关键因素给前端展示。代价是要维护两套逻辑，好处是模型该准就准、展示该懂就懂，两层互不干扰。你在比赛旁边看到的'法国整体实力占优''摩洛哥近期状态更佳'，就是这套展示层的输出。」
**过渡**：「第二个决策，是性能和部署的因果。」
**雷区**：不要展开 10 维特征是哪 10 个，说"原始特征"就够。

### P10 决策二：概率预计算 + 静态部署（~45 秒）
**讲什么**：把"算得快"和"敢做静态部署"串成因果。
**话术方向**：「第二个决策其实是一串因果。起点是性能墙：31 万次调用跑不动。我们用 91 组预计算砸开它，调用降到 496 次，两秒搞定。但这带来一个连带决策：既然算得这么快，我们敢做一个很激进的选择——纯静态部署。所有计算在 GitHub Actions 的 CI 里跑完，GitHub Pages 只发静态文件，零服务端依赖。换句话说，因为算得快，才敢把后端塞进 CI，前端纯静态。这是性能优化倒逼出来的架构决策。」
**过渡**：「但纯静态有个问题：不能交互。所以我们给了第二条路。」

### P11 决策三：双部署路径（~40 秒）
**讲什么**：两条路并存的产品思维。
**话术方向**：「部署我们给了两条路。第一条是 GitHub Pages，静态、每日自动重算、零成本，适合'看'。第二条是阿里云 Docker，FastAPI 四个端点，评委想'按一下让它重算'，POST 一下 /api/regenerate 就行，适合'玩'。Dockerfile 是多阶段构建，node:22 建前端、python:3.13 跑后端，单容器交付。CI/CD 是 10 步管线，push 到 main 就全自动。为什么两条都要？因为静态稳但不能交互，Docker 能交互但要运维——按场景给两条路，比只给一条更实在。」
**过渡**：「架构讲完。但这些决策不是一开始就想清楚的，是 8 轮迭代砸出来的。看工具链。」

## Act III · 工具链如何落地

### P12 八轮迭代地图（~30 秒 · 快带过）
**讲什么**：给开发地形图，30 秒带过，细节在后三页。
**话术方向**：「整个项目是 8 轮 WorkBuddy 会话迭代出来的，每轮之间用 handoff skill 交接上下文。我把它分成三段：前三轮打地基，中间三轮核心演进，最后两轮上线交付。我快速讲一下每段砸了什么墙。」
**过渡**：「先看打地基的三轮。」

### P13 第一阶段：从 0 到 1（轮 1-3）（~45 秒）
**讲什么**：代码、文档、数据三件事各一轮，三段式"问题→工具→突破"。
**话术方向**：「前三轮是打地基。第一轮，问题是从零搭全栈，我用 WorkBuddy 一次编排了 12 个 Task——7 个后端模块、5 个前端组件、加 CI/CD，一次会话产出可运行的全栈。第二轮，代码有了但缺文档，我用 Qoder RepoWiki 自动生成了 6 页架构文档和 3 张知识卡。第三轮是数据：训练集只有 117 场太少，我从 openfootball 拉了三届世界杯加两届欧洲杯，清洗后扩到 395 场。前三轮就干这三件事：代码、文档、数据。」
**过渡**：「地基打完，开始核心演进。」
**雷区**：不要逐个报 Task 名，"12 个 Task"是个量级感，不是清单。

### P14 第二阶段：核心演进（轮 4-6）（~50 秒）
**讲什么**：三个进化层层递进，技术含金量最高段。
**话术方向**：「中间三轮是技术含金量最高的。第四轮，问题是模拟还按全部 32 队跑，已淘汰的队伍污染预测。解法是把真实赛果注入，过滤到 14 支存活队，顺手把回测扩到 90 场，还用 Vite 分包把包体从 547K 压到 193K。第五轮，问题是只预测谁赢不够直观，我加了 Poisson 分布预测具体比分，每场给出像'巴西 2 比 1 挪威'这样的结果。第六轮，纯 Poisson 太保守，全是 1 比 0，我调了参数又加了平局修正，让比分多样化，2 比 1、0 比 2、3 比 0 都有。三轮进化一句话：先算对谁晋级，再算比分，最后让比分像真比赛。」
**过渡**：「最后两轮，收口交付。」
**雷区**：参数（decay 0.72→0.85）可以说，但别展开公式，说"调参+平局修正"就行。

### P15 第三阶段：上线交付（轮 7-8）（~40 秒）
**讲什么**：从能跑到能答辩到能交互，闭环。
**话术方向**：「最后两轮收口。第七轮，要参赛得有答辩材料，我通读全部源码写了完整的提交文档，覆盖架构、可视化、创新、加分项。第八轮，静态站不能交互，评委想按一下重算，我就上了 FastAPI 加 Docker，多阶段构建，部署到阿里云，Nginx 反向代理加 SSL 证书，CloudStudio 还能线上预览。至此从'能跑'到'能答辩'到'能交互'，闭环了。」
**过渡**：「工具链讲完。最后一个问题，也是评委最关心的——准吗？」

## Act IV · 结果有多准

### P16 冠军预测：法国 19.51%（~45 秒）
**讲什么**：展示有推理过程的结论。
**话术方向**：「先看冠军预测。法国 19.51% 排第一，巴西、摩洛哥紧随其后。但我想强调的不是这个数字，是它怎么来的。看法国的夺冠链路：16 强已经踢完，过了；8 强预测 3 比 0；4 强大约 65%；决赛大约 55%；最后夺冠大约 60%。你会发现，不是法国在某一步爆种，是它在每一步都'比较强'，概率一路乘下来最高。这才是蒙特卡洛的意义——它算的是路径，不是单点。」
**过渡**：「但'准不准'不能只看一个预测，得看回测。」

### P17 准确率证据：淘汰赛 88.9%（~50 秒 · 慢点）
**讲什么**：主动暴露小组赛 70%，用反差点制造记忆。
**话术方向**：「这是回测，四组数字。交叉验证 78.99%，综合 90 场 74.44%。我主动说一个不那么好看的：小组赛只有 70.83%。但有意思的是——淘汰赛反而高，18 场猜对 16 场，88.89%。为什么淘汰赛更准？因为淘汰赛有平局、加时、点球，模型处理这些的结构化优势更明显。而且这 90 场是独立验证集，是已经踢完的比赛作验证，不是训练集泄漏。所以我敢把 70.83% 也摆出来——一个敢把不完美亮出来的系统，才更值得信。」
**停顿点**：说完"才更值得信"后停 1.5 秒。
**过渡**：「最后看一个细节：它不只是给数字，它还会说人话。」

### P18 可解释性：Agent 会说人话（~35 秒）
**讲什么**：把决策一兑现成用户视角。
**话术方向**：「比如法国对摩洛哥这场，它不只给你一个 85% 对 5%，它还告诉你凭什么——法国整体实力占优、大赛底蕴更深、但摩洛哥近期状态更佳。评委可能看不懂随机森林，但一定看得懂'法国实力占优'。这就是可解释性的意义：Agent 要被信任，光准不够，还得让人懂。射手榜也是，姆巴佩已进 7 球、预测到 9 球，这种'已发生+预测'并列，比单纯一个数字有说服力。」
**过渡**：「到此，思考、架构、工具链、结果都讲完了。我做个收束。」

## 尾声

### P19 项目价值：这证明了什么（~40 秒）
**讲什么**：升维到"一种 Agent 工程范式"。
**话术方向**：「最后我想说，这不只是一个预测世界杯的项目。它证明三件事：第一，Agent 不是模型，是有边界的思考体，会记忆、会谦逊；第二，工程优化能让重计算变日常，10000 次模拟可以塞进 CI 每天跑；第三，可解释性不是附属功能，是 Agent 被信任的前提。如果非要给个公式：预测类 Agent = 记忆 + 想象 + 谦逊 + 人话 + 可验证。这是我交出的答案。」
**过渡**：「不信我说的话，上去按一下试试。」

### P20 线上体验 + Q&A（~30 秒）
**讲什么**：把验证权交给评委，开放姿态收尾。
**话术方向**：「两个入口都在屏幕上：GitHub Pages 可以直接看，阿里云 Docker 可以 POST 一下让它重新算。仓库也可以扫码。我的答辩到这里，欢迎提问，特别是'你凭什么信'这类问题——因为这个问题，我已经用 90 场回测和可解释推理链回答过了。谢谢。」
**雷区**：不要在 Q&A 页再总结一遍项目，留时间给提问。

## 答辩预判（Q&A 备弹）

| 可能的提问 | 回答方向 |
|-----------|---------|
| "85% 上限怎么定的？" | 经验值+防过度自信，可在 Q&A 说"试过 80 和 90，85 是准确率和保守度的平衡点" |
| "淘汰赛 88.9% 会不会样本太小？" | 承认 18 场样本不大，但强调是独立验证集，且会随赛事推进扩到更多场 |
| "为什么不 用 XGBoost/神经网络？" | 395 场数据量偏小，树模型更稳、可解释性更好；这是数据量与模型复杂度的取舍 |
| "蒙特卡洛 10000 次够吗？" | 14 队 91 组预计算后方差已收敛，做过收敛性检查；可加次数但收益递减 |
| "怎么保证不是过拟合？" | 训练集是历史 395 场，验证集是 2026 已完赛 90 场，时间上完全独立 |
